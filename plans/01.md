# OCIApp v1 Implementation Plan

## Summary

Build OCIApp as a three-package workspace for running customer applications packaged as OCI images and executed locally through a narrow UDS-based contract.

The system has three core deliverables:

- ociapp: the sandbox-side SDK and UDS server used inside application containers
- ociapp-build: the standalone build tool that packages application projects into .ociapp OCI archive artifacts
- ociapp-runtime: the host-side runtime that starts containers, communicates over UDS, maintains warm workers, and reclaims idle instances

The contract between host and application stays intentionally narrow:

- application code exposes a single execute function
- transport uses Unix domain sockets
- request and response transport payloads are opaque bytes
- transport envelopes are msgpack maps with stable request correlation

Implementation should stay compatible with future evolution where runtime management may move into a separate process or node-local agent without changing the service-to-
application contract.

## Key Changes

### Workspace structure

- Use the existing workspace root as the coordination point for all packages, shared development tooling, and validation commands.
- Expand the workspace to include packages/ociapp, packages/ociapp-build, and packages/ociapp-runtime.
- Keep package layouts consistent with the current repository conventions:
    - source under src/<package_name>/
    - tests under tests/python/
- Add shared development dependencies needed for implementation and validation:
    - pytest
    - pytest-asyncio
    - pytest-cov
    - mypy
    - prek
- Add runtime dependencies only where needed, with pydantic and msgpack as the primary shared protocol/modeling dependencies.
- Keep example projects optional support assets in-repo; they should document and validate the architecture but not add separate product surface area.

### ociapp

- Define the application-facing abstraction around a single execution entrypoint:
    - async execute(self, request: RequestT) -> ResponseT
- Provide an abstract Application[RequestT, ResponseT] base for user applications.
- Require each application to declare explicit request and response Pydantic models alongside the application instance so the runtime boundary is explicit and validated.
- Keep ociapp responsible for:
    - listening on a configured UDS path
    - receiving framed requests as bytes
    - decoding the outer transport envelope from msgpack
    - decoding inner payload bytes into a validated request model
    - invoking execute
    - dumping the response model to msgpack-compatible data
    - encoding the response payload back into bytes
    - returning the response envelope over the same socket
- Expose a CLI startup path:
    - ociapp serve --app pkg.module:app
- Support an optional socket-path override, but keep the container convention aligned to /run/ociapp/app.sock.
- Handle server lifecycle cleanly:
    - remove stale socket files on startup
    - create parent directories as needed
    - close sockets and clean up on shutdown
- Separate internals into:
    - application loading
    - protocol framing
    - envelope serialization
    - payload model serialization
    - server lifecycle
    - CLI entrypoint
- Reuse this protocol implementation directly in ociapp-runtime to keep client and server wire-compatible by construction.

### ociapp transport contract

- Use a simple framed protocol over UDS:
    - 4-byte big-endian length prefix
    - followed by msgpack-encoded envelope bytes
- Keep request and response bodies as msgpack maps.
- Use these stable transport envelopes:

{
  "request_id": UUID,
  "payload": bytes,
}

{
  "request_id": UUID,
  "payload": bytes | None,
  "error": bytes | None,
}

- Treat the inner payload as opaque transport bytes from the host/runtime perspective.
- For sandbox-side model execution:
    - decode payload bytes from msgpack into user request data
    - validate into the declared request model
    - serialize the response model into msgpack bytes for the response payload
- Encode application errors into error as msgpack bytes for a machine-readable structure containing at least:
    - error type
    - message
    - optional details
- Reject malformed frames, malformed envelopes, and mismatched request/response correlation data deterministically.

### ociapp-build

- Implement ociapp-build as a standalone CLI tool.
- It must not replace, wrap, or interfere with the project’s declared Python build backend.
- Read target project configuration from [tool.ociapp-build] in the target project’s pyproject.toml.
- Make the documented primary CLI invocation:
    - ociapp-build /path/to/project/root
- Optional flags may be added for output control, but the positional project-root form remains the canonical interface.
- Produce a physical local OCI archive artifact and never require a registry.
- Output filename convention:
    - {project_name}-{version}.ociapp

#### Build modes

- Support exactly two modes.

Managed mode:

- This is the default mode.
- Inputs:
    - canonical Python project metadata from pyproject.toml
    - entrypoint
    - optional system-packages
- Behavior:
    - generate a temporary built-in Containerfile
    - inject requested system packages through build args or equivalent generated configuration
    - build the target project into a wheel
    - install that pre-built wheel into the image
    - install ociapp into the image as needed
    - do not copy raw application source into the final image as the primary install mechanism
    - set container ENTRYPOINT to ["tini", "--"]
    - set container CMD to ["ociapp", "serve", "--app", "<entrypoint>"]

Custom mode:

- Enabled when mode = "custom".
- Only containerfile is permitted in this mode.
- Delegate image construction directly to the provided Containerfile.
- Perform no validation that the resulting image is OCIApp-compliant.

#### ociapp-build implementation shape

- Separate:
    - config parsing and validation
    - project metadata extraction
    - wheel build orchestration
    - temporary build-context generation
    - container-engine command execution
    - archive export
- Use Podman commands directly in v1.
- Keep command execution behind a thin abstraction so unit tests can validate command construction without depending on a real engine.
- Fail fast on:
    - invalid config combinations
    - missing entrypoint in managed mode
    - nonexistent containerfile in custom mode
    - unsupported extra keys in custom mode

### ociapp-runtime

- Implement ociapp-runtime as the host-side execution engine used by services.
- Keep the primary public API transport-agnostic and bytes-first:

async with Runtime(...) as runtime:
    result = await runtime.execute(image_path, payload: bytes) -> bytes

- The runtime is responsible for:
    - loading or unpacking a local .ociapp artifact
    - starting container instances
    - mounting a host-visible runtime directory into each container
    - waiting for /run/ociapp/app.sock to become available
    - connecting to the container UDS
    - framing requests
    - msgpack envelope encoding/decoding
    - request correlation
    - timeout enforcement
    - response parsing
    - warm-pool lifecycle management
- Keep typed payload decoding as a sandbox-side concern of ociapp; host-side typed helpers are out of scope for the core v1 API.

#### Runtime model

- Treat each container instance as a long-lived worker.
- Manage instances per image artifact.
- For each image:
    - keep zero or more warm instances
    - dispatch requests to an available instance
    - stop instances after the configured idle timeout
    - stop all instances on runtime shutdown
- Track instance state using:
    - starting
    - ready
    - busy
    - stopping
- Each instance must also track:
    - last-used timestamp
    - active request count
- An instance is eligible for idle shutdown only when:
    - it is ready
    - it has no active requests
    - idle time exceeds the configured threshold

#### Runtime implementation shape

- Use Podman as the only supported engine in v1.
- Split internal responsibilities into:
    - artifact loading
    - container engine adapter
    - UDS client protocol
    - pool and instance management
    - public runtime API
- Use a dedicated runtime manager task or thread to own mutable instance state and background idle reaping.
- Keep the engine boundary narrow so the architecture can later support moving runtime management into a separate process without changing public behavior.
- Reuse the same framing and envelope code as ociapp so protocol compatibility remains mechanically guaranteed.

## Public APIs and interfaces

### ociapp

- Application[RequestT, ResponseT]
- async execute(self, request: RequestT) -> ResponseT
- application descriptor or equivalent registration object that binds:
    - app instance
    - request model
    - response model
- CLI:
    - ociapp serve --app <import-path> [--socket-path <path>]

### ociapp-build

- CLI:
    - ociapp-build <project-root>
- [tool.ociapp-build] configuration fields:
    - mode
    - entrypoint
    - system-packages
    - containerfile

### ociapp-runtime

- Runtime
- async context management for startup and shutdown
- await runtime.execute(image_path, payload: bytes) -> bytes

## Examples

- Include an example application project that demonstrates:
    - request and response Pydantic models
    - an Application implementation
    - tool.ociapp-build configuration
    - successful managed-mode packaging
- Include an example runtime consumer that demonstrates:
    - loading a built .ociapp
    - sending at least two requests through the runtime
    - observing warm instance reuse
    - observing idle shutdown after inactivity
- Keep examples as reference implementations only; they should not define new required APIs.

## Test Plan

### ociapp

- framing round-trips for valid requests and responses
- malformed frame rejection
- envelope encode/decode behavior
- request/response request_id correlation handling
- request payload decode and Pydantic validation success
- request payload validation failures
- response model serialization to payload bytes
- application exception mapping into structured error payloads
- stale socket replacement on startup
- UDS server lifecycle and clean shutdown
- CLI import-path loading behavior

### ociapp-build

- config parsing for valid managed and custom cases with explicit parameterized case ids
- invalid config combinations
- missing required fields
- nonexistent custom Containerfile handling
- project metadata extraction for output filename derivation
- managed Containerfile generation
- wheel-oriented managed build behavior
- system-package injection behavior
- Podman command construction for build and archive export
- process execution mocked so tests do not require Podman

### ociapp-runtime

- image-keyed warm-pool behavior
- instance state transitions across startup, ready, busy, and stopping
- startup wait logic for socket readiness
- request dispatch to warm instances
- request timeout handling
- startup timeout handling
- shutdown timeout behavior
- malformed or mismatched response rejection
- request correlation enforcement
- idle reaping behavior based on state, active count, and timestamps
- Podman adapter command construction and cleanup sequencing

### Integration-style validation

- in-process compatibility tests between ociapp server and ociapp-runtime client without real containers
- example-project smoke tests validating configuration and command wiring
- optional real-container integration tests may be isolated behind markers if Podman is not always available in local or CI environments
- all pure-Python behavior must still have deterministic unit coverage

### Required validation commands

- uv run pytest --cov
- uv run mypy .
- prek run -a

## Assumptions and defaults

- Python type conventions should follow the repository guidance for Python 3.13+ style typing.
- Public-facing functions and classes should include concise docstrings.
- Unit tests should follow the repository’s pytest layout and style conventions.
- Podman is the only supported container engine in v1.
- .ociapp is a local OCI archive tar artifact, not a registry reference.
- The runtime-to-application contract remains UDS plus framed msgpack envelopes carrying opaque payload bytes.
- ociapp-build is intentionally CLI-only in v1.
- Example projects live in-repo as validation and documentation assets.
- Any future host-side typed convenience layer is out of scope unless explicitly added as a separate planned feature.
