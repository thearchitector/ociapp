# Request-ID Multiplexing and Structural Concurrency

## Summary

- Replace the current one-request-per-socket client in
  [client.py](/home/elias/Codespace/ociapp/packages/ociapp-runtime/src/ociapp_runtime/client.py#L25),
  the one-request-per-worker scheduling in
  [runtime.py](/home/elias/Codespace/ociapp/packages/ociapp-runtime/src/ociapp_runtime/runtime.py#L145),
  and the serial per-connection server loop in
  [server.py](/home/elias/Codespace/ociapp/packages/ociapp/src/ociapp/server.py#L90)
  with true request-ID multiplexing over one persistent UDS connection per warm
  worker.
- Keep the wire format unchanged. `request_id` becomes the live correlation key
  for multiple in-flight requests on the same socket, and responses may be
  written and received in completion order rather than send order.
- Preserve the public `Runtime.execute(image_path, request) -> dict[str, object]`
  API. The behavior change is that concurrent `execute()` calls for the same
  artifact share one warm worker connection instead of forcing per-request
  sockets or extra workers.

## Implementation Changes

### Runtime and client

- Introduce an internal persistent worker-session object that owns one
  `StreamReader`, one `StreamWriter`, a write lock, and a
  `pending: dict[UUID, Future[dict[str, object]]]`.
- Add a background response-reader task per worker that continuously reads
  frames, decodes response envelopes, matches by `request_id`, and resolves the
  correct pending future regardless of arrival order.
- Handle response outcomes as follows:
  - matched success response: decode payload and resolve only that request
  - matched remote error response: decode error and fail only that request
  - matched response whose inner payload or error body is invalid: fail only
    that request with `ResponseProtocolError`
  - unknown `request_id` or late response for a timed-out request: drop it,
    keep the worker alive, and leave other requests untouched
  - connection close, frame parse failure, or response-envelope decode failure:
    fail all pending requests on that connection and retire the worker
- Move per-request timeout handling into the shared-session call path. On
  timeout, remove that request from `pending`, raise `RequestTimeoutError` only
  to that caller, and keep the worker alive; any later response for that
  `request_id` is treated as stale and dropped.
- Rework scheduling so a worker with active requests is still dispatchable. Keep
  `active_request_count`, but stop treating `InstanceState.BUSY` as exclusive.
- Coalesce cold-start concurrency per image: if a non-stopping worker already
  exists in `STARTING`, `READY`, or `BUSY`, new callers wait for or dispatch
  onto that worker instead of starting another container. Start a new container
  only when the pool has no usable worker.
- Keep idle reaping based on `active_request_count == 0` and `idle_timeout`. A
  worker with pending requests is never reapable.
- Keep no-retry semantics for transport failure: unexpected connection loss
  fails all pending callers and the next demand creates a replacement worker.

### Structural concurrency and lifecycle

- Back `Runtime.start()` with a runtime-scoped `AsyncExitStack` plus one entered
  `asyncio.TaskGroup`.
- Register the reaper task and each worker response-reader task in that task
  group so they share one lifecycle boundary.
- Register socket close, pending-future fail-fast cleanup, container stop, and
  mount-dir removal through the exit stack so `Runtime.close()` has one
  structured teardown path.
- `Runtime.close()` becomes fail-fast: reject new work, fail all pending
  requests immediately, close shared sockets, then stop containers and remove
  runtime directories.

### Server

- Rework server connection handling so a single client connection can pipeline
  multiple requests:
  - keep reading frames without waiting for earlier requests to finish
  - start one task per decoded request
  - serialize writes with a per-connection write lock
  - let responses go out in task-completion order, not request-read order
- Keep one response per request and preserve the existing error-envelope
  behavior for validation failures and application exceptions.
- If the server hits a connection-level protocol failure while reading from a
  pipelined client, close the connection and cancel outstanding request tasks
  for that connection.

## Public Interfaces

- No wire-shape change: `RequestEnvelope` and `ResponseEnvelope` stay as-is.
- No required public API addition: `Runtime.execute(...)` remains the caller
  entry point, and this change does not add a per-worker concurrency knob
  because the chosen policy is uncapped pipelining on a single warm worker.
- `InstanceState` remains exported, but `BUSY` now means "has active in-flight
  requests" rather than "cannot accept more work."

## Test Plan

- Add a server test that sends two requests over one fake connection, makes the
  second finish first, and asserts the responses are emitted out of order but
  still correlated by `request_id`.
- Add a runtime test that issues concurrent `Runtime.execute()` calls for one
  artifact, asserts only one worker/container is started, and verifies both
  calls complete correctly when responses arrive out of order.
- Add a cold-start test that concurrent first requests coalesce onto one
  `STARTING` worker instead of spawning duplicates.
- Add client/session tests for per-request timeout isolation, late-response
  dropping, and unknown `request_id` dropping without retiring the worker.
- Add fatal-transport tests showing that connection loss, truncated frames, or
  invalid response envelopes fail every pending request and retire the worker.
- Add shutdown tests showing `Runtime.close()` immediately fails in-flight
  requests and still stops the container exactly once.
- Keep and update the existing warm-reuse, idle-reap, remote-error, and
  startup-timeout coverage to reflect persistent shared connections rather than
  one-shot sockets.

## Assumptions and Defaults

- The target is true single-socket pipelining per worker, not multiple
  concurrent connections to the same container.
- Unmatched or stale responses are ignored only when the outer frame and
  envelope are still decodable. Malformed framing or malformed response
  envelopes remain fatal because the shared stream can no longer be trusted
  after decode failure.
- No automatic retry is performed for failed or dropped shared connections, to
  avoid duplicate execution of non-idempotent application logic.
