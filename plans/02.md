# Docker Cutover and Transport Simplification Plan

## Summary

Switch the workspace from Podman to Docker, using Docker Buildx as the only
supported build and export path for `.ociapp` artifacts.

At the same time, reduce code footprint in touched areas by:

- removing production `assert` usage from source files
- collapsing one-line wrappers and duplicate protocols
- shrinking `ociapp-build` public exports to the supported surface
- replacing untrusted cross-boundary transport records with frozen Pydantic
  models

The transport boundary between runtime and container should treat request,
response, and structured error envelopes as validated wire models rather than
lightweight tuple-like records.

## Key Changes

### Docker cutover

- Replace `PodmanAdapter` with `DockerAdapter`.
- Update `Runtime` to default to `DockerAdapter`.
- Remove Podman-specific wording from mutable docs, READMEs, tests, and
  examples.
- Keep `EngineAdapter` as the single runtime engine protocol and delete the
  duplicate `RuntimeEngine` protocol from `runtime.py`.

### Build path

- Replace the current `build` plus `save --format oci-archive` flow with a
  single Docker Buildx command in both managed and custom modes:

```bash
docker buildx build \
  --tag <image_tag> \
  --file <containerfile> \
  --output type=oci,name=<image_tag>,dest=<artifact_path>,tar=true,compression=zstd,compression-level=22 \
  <context>
```

- Delete the separate archive export helper rather than reintroducing it under a
  different name.
- Keep the `.ociapp` filename convention unchanged.

### Runtime assertions and structural cleanup

- Remove every production `assert` from `packages/*/src`.
- In `packages/ociapp-build/src/ociapp_build/build.py`, avoid type assertions by
  passing already-narrowed config objects into helper functions.
- In `packages/ociapp-runtime/src/ociapp_runtime/runtime.py`, replace the cached
  image-reference assertion with explicit non-optional control flow.
- In `packages/ociapp/src/ociapp/server.py`, replace the server handle assertion
  with an explicit lifecycle error.
- Inline trivial wrappers in `ociapp.cli`, `ociapp.server`, and
  `ociapp_runtime.runtime` when they only forward to a single function or
  stdlib call.

### Public surface reduction

- Keep `build_project()` as the supported `ociapp-build` entry point.
- Stop re-exporting internal build helpers such as:
  - `build_image_tag`
  - `build_wheel`
  - `prepare_managed_context`
  - `resolve_artifact_path`
  - `render_managed_containerfile`
- Update tests to target supported entry points or module-local helpers instead
  of package-root re-exports where needed.

## Transport Model Changes

### Pydantic transport records

- Replace `RequestEnvelope`, `ResponseEnvelope`, and `ErrorPayload` with frozen
  Pydantic models.
- Keep their public names unchanged so downstream code still imports:
  - `ociapp.RequestEnvelope`
  - `ociapp.ResponseEnvelope`
  - `ociapp.ErrorPayload`

Use these fields:

- `RequestEnvelope`
  - `request_id: UUID`
  - `payload: bytes`
- `ResponseEnvelope`
  - `request_id: UUID`
  - `payload: bytes | None`
  - `error: bytes | None`
- `ErrorPayload`
  - `error_type: str`
  - `message: str`
  - `details: object | None`

### Validation behavior

- Encode the response-envelope invariant in a Pydantic model validator so
  exactly one of `payload` or `error` is present.
- Keep validation at the codec boundary:
  - `decode_request_envelope()` should unpack msgpack and then call
    `RequestEnvelope.model_validate(...)`
  - `decode_response_envelope()` should unpack msgpack and then call
    `ResponseEnvelope.model_validate(...)`
  - `decode_error_payload()` should unpack msgpack and then call
    `ErrorPayload.model_validate(...)`
- Translate any Pydantic `ValidationError` raised during transport decoding into
  `ProtocolError`.
- Encode transport records with `model_dump(mode="python")`, converting
  UUID-containing fields to the existing wire shape expected by the protocol.

### Non-transport immutable records

- Keep immutable, non-transport pass-through values lightweight.
- Convert both build/runtime `CommandResult` types to `NamedTuple`.
- Replace the dataclass-based `RemoteExecutionError` with a normal exception
  class that stores an `error: ErrorPayload` attribute.
- Keep mutable runtime state holders such as `WorkerInstance` and `ImagePool`
  mutable, because they are updated during runtime lifecycle management.

## Expected Public API Changes

- `ociapp_runtime.DockerAdapter` becomes the public runtime adapter.
- `ociapp_runtime.PodmanAdapter` is removed.
- `Runtime(engine=...)` still accepts any implementation matching
  `EngineAdapter`.
- `ociapp.RequestEnvelope`, `ociapp.ResponseEnvelope`, and
  `ociapp.ErrorPayload` remain public names but become frozen Pydantic models.
- `ociapp_build` no longer exposes internal helper functions from the package
  root.

## Test Plan

- Update build tests to assert one Docker Buildx command per build path and the
  absence of a separate save/export command.
- Update engine tests to assert `docker load`, `docker run`, and `docker stop`
  command construction and Docker-specific failure messages.
- Update protocol tests so request, response, and error payload failures are
  exercised through Pydantic-backed decoding and still map to `ProtocolError`.
- Add direct tests for `ResponseEnvelope` validation covering:
  - both `payload` and `error` missing
  - both `payload` and `error` present
- Update CLI tests to patch `load_application()` and `serve_application()`
  directly after wrapper removal.
- Add checks for the renamed adapter export and the pruned `ociapp-build`
  package-root exports.
- Run the full required verification after implementation:
  - `timeout 180s uv run pytest --cov`
  - `timeout 300s uv run mypy .`
  - `timeout 600s uv run prek run -a`

## Assumptions

- Docker CLI and Buildx are required; Podman support is removed rather than
  retained behind compatibility aliases.
- Frozen Pydantic models are preferred for cross-boundary transport data because
  these structures are untrusted and benefit from built-in validation.
- `ARCHITECTURE.md` and `plans/01.md` remain unchanged because they include
  `pragma: no ai`.
